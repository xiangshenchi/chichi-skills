# Skill Article Writer

根据平台规则 URL，自动生成符合要求的 Skill 介绍文章（参赛稿、推广帖、投稿等）。

---

## 一、整体流程

```
① 抓取规则 URL  →  ② 解析平台要求  →  ③ 收集 Skill 信息  →  ④ 生成文章  →  ⑤ 输出/导出
```

---

## 二、Step 1：抓取并解析规则页面

用 `web_fetch` 获取用户提供的 URL 全文，然后提取以下信息：

### 2.1 必须提取的内容

| 提取项 | 说明 |
|--------|------|
| **标题格式** | 是否有固定前缀，如`【Skill 创作】`、`[投稿]`等 |
| **正文结构** | 官方要求的章节顺序和名称（如：简介/场景/过程/步骤/效果/链接/总结） |
| **各章节说明** | 每个章节要求写什么，有没有示例 |
| **字数/篇幅要求** | 有无最低字数、最大字数 |
| **禁止事项** | 哪些内容会被判为无效投稿 |
| **加分项** | 哪些内容会提升获奖/过审概率 |
| **附件规则** | 图片、视频、压缩包是否可以上传，还是只能用外链 |
| **截止时间** | 投稿时间区间 |
| **标签/分类** | 发帖时需要选择什么标签 |

### 2.2 解析后，向用户简要确认

用一段话总结你理解的平台要求，格式如下：

> 📋 **规则摘要**
> - 平台：XXX
> - 标题格式：`【XXX】+ 描述`
> - 正文结构：简介 → 场景 → 过程 → 步骤 → 效果 → 链接 → 总结（共 7 节）
> - 核心评分维度：实用性、过程完整度、有真实截图
> - 禁止：纯 AI 生成内容、无实际体验
> - 附件：不支持上传压缩包，需用外链
>
> 理解无误的话，请提供你的 Skill 信息 👇

---

## 三、Step 2：收集 Skill 信息

如果用户在触发时已经提供了 Skill 信息（在对话历史里），**直接从对话中提取**，不要重复询问。

如果信息不完整，按优先级询问（一次问完，不要一条一条问）：

### 必须知道的（缺一不可）

- **Skill 名称**：叫什么
- **解决的问题**：它替代了什么繁琐的手工步骤
- **核心能力**：1~3 句话说清楚它做什么
- **真实使用场景**：你在什么情况下用到了它，最好有具体例子

### 锦上添花的（有就更好）

- **使用前 vs 使用后的效率对比**：具体数字（时间、步骤数等）
- **最让你意外/印象最深的输出**：某次它发现了你自己没注意到的问题
- **目前的局限性**：还不完善的地方
- **后续优化方向**：想怎么改进
- **Skill 文件链接 / GitHub / 文档地址**

---

## 四、Step 3：生成文章

### 4.1 严格遵循平台模版结构

按照 Step 1 提取的章节顺序生成，**不要自作主张增减章节、改变顺序**。

如果某章节在规则里有示例，输出风格要参考示例的语气和颗粒度。

### 4.2 各章节写作要点

#### 【简介 / Skill 介绍】
- 一两句话：叫什么名字、解决什么问题、适合谁用
- 语言简洁，让不了解背景的人也能秒懂
- 不要把使用步骤放在这里

#### 【使用场景】
- 必须写**真实的、具体的**场景，不能泛泛说"适合所有人"
- 回答三个问题：你为什么想做它？之前遇到什么麻烦？做出来能省掉哪些动作？
- 如果用户提供了真实痛点故事，直接用，效果比模板话好

#### 【创作过程】（通常是核心章节，要写得最详细）
- 展示思考逻辑：为什么这么设计、踩过什么坑
- 展示关键实现：可以贴提示词片段、工作流结构、配置说明
- **不要只放结果**，评审要看到"你是怎么把它做出来的"
- 如果有有趣的设计决策（比如"为什么选择固定7个Part而不是动态生成"），写出来

#### 【使用步骤】
- 清晰的操作流程，Step 1 / Step 2 / Step 3 格式
- 包含：如何触发、输入什么、得到什么输出
- 如果有多种触发方式，都列出来

#### 【效果展示】
- 用真实数据，不要虚构例子
- 有"使用前 vs 使用后"对比最好
- 如果有让人意外的发现/输出，重点展示——这是最有说服力的部分
- 提醒用户：发帖时这里需要配截图

#### 【Skill 链接 / 附件】
- 按平台规则格式放链接
- 如果平台不支持上传压缩包，提示用户用网盘外链

#### 【总结与思考】
- 效率提升了多少（具体数字）
- 目前最满意的地方是什么
- 后续想怎么优化
- 最后一两句真实感悟（不要空话）

### 4.3 写作风格原则

- **真实优先**：用用户提供的真实数据、真实场景，不要填充假例子
- **有个人判断**：不是流水账，要有"为什么这样设计"的思考
- **避免 AI 味**：平台明确禁止"只靠 AI 拼接、没有真实体验"，语气要口语化、有温度
- **标题抓人**：在符合平台格式的前提下，标题里尽量带具体数字或结果（如"1小时→2分钟"）

---

## 五、Step 4：输出与导出

### 5.1 默认输出
直接在对话中输出完整 Markdown 文章，用户可以直接复制粘贴到平台。

### 5.2 如果用户需要文件
说"帮我导出 Word 版本"或"帮我保存成文件"，则调用 `docx` skill 或直接生成 `.md` 文件。

### 5.3 输出后的提示
文章末尾附上一个简短的发帖 checklist：

```
📝 发帖前检查：
□ 标题格式是否符合平台要求（如加了【Skill 创作】前缀）
□ 选了正确的标签/分类
□ 效果展示部分是否有截图（文章里只有文字，截图需自己上传）
□ Skill 链接是否已替换为真实外链
□ 如平台不支持压缩包，是否已改为网盘链接
```

---

## 六、多平台适配说明

此 skill 通用，不绑定特定平台。遇到常见平台类型的处理方式：

| 平台类型 | 重点 |
|----------|------|
| **技术比赛/征集**（如 TRAE SOLO 创作赛） | 重点写创作过程和真实效果，评审看过程完整度 |
| **技术社区**（如 CSDN、掘金） | 重点写技术实现细节，读者喜欢看提示词/配置 |
| **内容社区**（如 小红书、小黑盒） | 重点写痛点场景和效果对比，语气要口语化 |
| **个人博客** | 结构最自由，可以加更多背景思考和感悟 |

如果 URL 对应的是未知平台，按正常流程抓取规则，从规则内容判断平台风格。

---

## 七、缺失信息处理

| 缺失内容 | 处理方式 |
|----------|----------|
| 用户没给 URL | 询问："你希望投稿到哪个平台？请提供规则页面链接" |
| URL 无法访问 | 提示用户，请求手动粘贴规则文本 |
| 规则页面信息不全 | 用抓到的内容尽量生成，并在文章开头注明"以下结构基于现有规则，请核对平台最新要求" |
| Skill 信息不完整 | 一次性列出所有缺失项询问，生成文章时对缺失部分标注`[待补充：XXX]` |

---

## 八、快速示例

**用户说**："你能参考这个网站里的信息和规定 https://xxx.com/rules 帮我写一篇参赛文章吗"

**处理流程**：
1. `web_fetch` 抓取该 URL
2. 提取平台规则，输出规则摘要确认
3. 检查对话历史中是否已有 Skill 信息
4. 如有 → 直接生成文章；如无 → 一次性询问必要信息
5. 按平台模版生成完整文章
6. 输出发帖 checklist
