# 我给 Claude 装了个「外挂」，现在训练完 YOLO 模型它会自动帮我写分析报告

> 适合平台：个人博客 / 小黑盒 / CSDN  
> 标签：#Claude #AI工具 #YOLO #深度学习 #目标检测 #效率工具

---

## 起因：每次训完模型都要手动写一堆东西

做毕设的时候，每跑完一次 YOLO 训练，我都要做这些事：

- 打开 `results.csv`，找最佳 epoch 是哪个
- 看 loss 曲线有没有过拟合迹象
- 把 per-class 的 mAP50、Recall 抄到文档里
- 看混淆矩阵，分析哪个类容易混淆
- 对着数据想改进方向，整理成文字

这套流程我至少跑了七八次，每次都要花一个多小时。数据都在，就是苦于把它们**变成可读的分析报告**这件事太费时间。

然后我想到了一件事：Claude 不是支持安装自定义 Skill 吗？

---

## 什么是 Claude Skill？

简单说就是给 Claude 装「插件」，告诉它遇到某类任务时用什么固定流程来处理。Skill 本质是一个 Markdown 文件，写清楚触发条件、处理步骤、输出格式，存到指定目录后 Claude 就会自动识别并调用。

我花了一下午，结合自己的 YOLO 训练输出规律，写了一个专门的 Skill：`yolo-report-analyst`。

---

## 它能干什么

把 YOLO 训练完后输出的这些文件丢给 Claude：

```
results.csv          ← 每个 epoch 的 loss 和 mAP 数据
args.yaml            ← 训练配置
训练日志 (.md/.log)  ← 完整训练过程
confusion_matrix_normalized.png  ← 混淆矩阵图
（以上任意组合都行，缺文件它会降级处理）
```

然后说一句「帮我分析训练结果」，它会自动输出一份包含 7 个 Part 的完整报告：

| Part | 内容 |
|------|------|
| 1 | 训练配置概览表格 |
| 2 | 最终性能指标 + per-class 逐类分析 |
| 3 | 关键 epoch 节点表 + loss 曲线诊断 |
| 4 | 混淆矩阵量化解读 |
| 5 | 推理效率分析（FPS、延迟分解） |
| 6 | 问题诊断 & 改进建议（带原因和具体方案） |
| 7 | 总结 + 优先级行动计划 |

如果你需要 Word 版本，说一声它会直接生成 `.docx` 文件供下载。

## 真实效果展示

这是我用自己的绝缘子缺陷检测项目跑出来的，两次训练（3类版本 vs 8类版本）都扔进去，让它分别分析 + 横向对比。

**Run1（3类，insulator_mvp_150ep）的 per-class 分析表：**

| 类别 | Precision | Recall | mAP50 | mAP50-95 |
|------|-----------|--------|-------|----------|
| insulator | 0.997 | 1.000 | 0.995 | 0.968 |
| flashover | 0.853 | 0.757 | 0.836 | 0.367 |
| broken | 0.878 | 0.885 | 0.904 | 0.569 |
| **综合** | **0.909** | **0.881** | **0.912** | **0.635** |

这些数字它不只是列出来，还会告诉你：

> "flashover 的 Recall 仅 75.7%，约 1/4 的闪络事件被漏检。混淆矩阵显示 19% 被误判为背景——原因是单图实例密集（68张图里230个实例），小目标漏检风险高。建议启用 copy_paste=0.3 数据增强，专项针对该类。"

**比最让我惊喜的**是它在对比两次实验时，发现了我自己没注意到的问题：Run2 把 batch 从 16 扩到 32 + 双 GPU，但 AdamW **自动把学习率从 0.001429 降到了 0.000833**，低了 42%，这是两次实验性能差距的关键原因之一。我之前完全没想到去查这个。

## Skill 的核心设计思路

分享几个我在写这个 Skill 时想清楚的点，供参考：

**① 触发词要覆盖自然语言的多种说法**

不能只写「当用户说'分析训练结果'时」，因为实际使用时你可能说「看看效果怎么样」「帮我总结一下」「给我写个报告」。Skill 的 description 要把这些都覆盖到。

**② 缺文件时要有降级策略，而不是直接报错**

训练输出不是每次都完整的。Skill 里写清楚：没有 results.csv 就只用日志；没有混淆矩阵就跳过 Part 4；仅有图片就做视觉描述分析。始终输出能输出的部分。

**③ 报告结构要固定，便于日后批量对比**

每次都是 7 个 Part、相同顺序，这样多次训练的报告放在一起很容易横向比较，不会因为每次格式不同而没法对比。

**④ 最有价值的部分是「问题诊断」，不是指标列表**

指标列出来只是原材料，真正有用的是「这个数字意味着什么、为什么会这样、怎么改」。Skill 要求每条改进建议必须包含：观察到的现象 → 可能原因 → 具体方案，不能泛泛而谈。

## 如何安装使用

Skill 文件就是一个 Markdown 文件，放到 Claude 的 skills 目录就能用。

我把打包好的 Skill 放在下面，解压后是这个结构：

```
yolo-report-analyst/
└── SKILL.md
```

把 `yolo-report-analyst/` 文件夹放到你的 skills 目录（通常是 `/mnt/skills/user/`）即可。之后只要上传训练输出文件，说「分析一下我的训练结果」，Skill 就会自动触发。

---

## 适合什么人用

- 在用 YOLO（v5/v8/v11/ultralytics 系列）做目标检测的
- 需要写毕业论文、项目报告、实验记录的
- 跑了多次训练实验，需要系统整理对比的
- 懒得每次手动分析 loss 曲线和混淆矩阵的（就是我）

不适合的情况：如果你的训练任务特别简单，或者已经有成熟的实验管理平台（MLflow/W&B），可能用处没那么大。

---

## 最后

这个 Skill 目前基于我自己的 YOLO 训练输出格式写的（Ultralytics 8.x），理论上也适用于同类格式的其他版本。如果你有不同格式的训练输出（比如 PaddleDetection、MMDetection），Skill 里的解析逻辑可能需要微调。
